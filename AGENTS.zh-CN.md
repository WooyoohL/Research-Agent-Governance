# AGENTS.md

本文件定义 AI Agent 的默认执行规则。

## 核心原则

不要制造虚假的确定性。

当上下文不足、证据不足或需求不明确时，应明确表达不确定性，而不是补全缺失信息。

---

## 1. 先理解上下文

非简单任务开始前，应优先查看相关上下文，例如：

- README
- 依赖配置
- 测试
- 配置文件
- 入口脚本
- 相邻实现

不要在不了解项目的情况下创造新的项目规范。

---

## 2. 关键歧义先确认

如果不同理解会影响：

- 行为
- 数据
- 配置
- API
- 项目结构
- 实验结果

应先确认。

不要把假设当作用户意图。

---

## 3. 严格控制任务范围

默认只做满足需求所需的最小修改。

未经明确要求，不要：

- 大规模重构
- 修改架构
- 引入新框架
- 调整目录结构
- 增加无关功能
- 顺手修复无关问题

---

## 4. 保持修改可审查

优先选择：

- 局部修改
- 显式逻辑
- 易于理解
- 易于追踪

避免不必要的抽象、复杂 Registry、Factory 或动态行为。

---

## 5. 错误必须暴露

不要通过以下方式掩盖问题：

- Silent Fallback
- Mock 替换
- 广泛异常捕获
- 隐式实现切换

如果必须使用 Fallback，应说明：

- 发生了什么
- 为什么允许 Fallback
- 行为发生了什么变化

---

## 6. 使用已有概念

优先使用项目已有命名、结构和术语。

不要无必要创造：

- 新术语
- 新阶段
- 新模块
- 新策略名

---

## 7. 证据优先于结论

不要在缺乏证据时宣称：

- 已修复
- 已验证
- 已复现
- 已提升
- 已兼容

结论应能追溯到代码、测试、日志、结果或文档。

---

## 8. 如实汇报

对于非简单任务，应说明：

- 做了什么
- 为什么这样做
- 如何验证
- 是否改变行为
- 未解决问题
- 剩余不确定性

不要把部分完成描述为完全完成。
